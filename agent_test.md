### 测试过程

  目标是基于pump实现一个简单的异步rpc框架.要求可以静态注册和分配服务,并且在编译期完成.同时要求代码的质量和效率
  
  首先使用junie完成了一系列的简单任务,作为实现一个rpc框架的前置.然后使用junie实现rpc服务.
  
  之后在等待cc账号的时候手写了一版.
  
  最后使用 claude code 完成了一版.

### 测试总结

#### 简单任务:
    
  类似增加'when_any'算子,完善'net scheduler'这类的任务,junie就完成很好,基本上可以直接用.以至于后续手写和claude code直接采用了junie对
  简单任务的提交.

#### rpc框架,

##### junie
        
  实现的不好.从细化需求开始,就发现需求文档过大,导致迭代版本增加需求(比如要求静态注册和分配服务)后,文档的前后自相矛盾.完成代码的时候发现拼凑代码的情况比较严重.细节有点惨不忍睹.甚至示例程序和rpc框架关系不大.有明显的测试不通过硬去凑代码的感觉.

  但是在沟通过程中,实际去问他对某些东西的理解.又发现他的理解大部分其实是正确的.不知道是不是因为我提示词的问题引起了这种情况.

##### claude code
          
  实现的不错,代码功能相当完整,实现了许多功能以及边界条件的处理.比如链路双向复用等等.做到了服务在编译期静态注册和分配.对于代码效率的处理也算不错.
  清晰准确的理解了那些代码需要做并发处理,哪些不需要.该用thread local绝不用原子,能用单原子的时候绝不用多原子.知道怎么写逻辑以适配对效率的要求

  我个人觉得问题在于对框架代码整体的设计和我的想法有些不同,这点有点难描述,就是我觉得设计出来应该是这个样子的,他实现出来时那个样子的,具体可以看我手写代码的区别.
  
  这点导致rpc框架的实现逻辑有点散.这里一段那里一段.结果上导致示例程序的client逻辑里多了一个很突兀的dispatch循环.这和我设计pump库的初衷有点违背
  但是这点又从最终实现效果上挑不出什么问题,该考虑到的也都考虑到了,该处理的也都处理了.只是不"美"而已.但是这个"美"有点见仁见智.

### 最终个人结论:

  对于简单任务,差别不大.特别是有类似的代码的任务,比如有'when_all'算子的情况下,实现'when_any'算子.都可以完成的很好
  
  对于复杂任务,使用junie和claude code的差别比较大.甚至因为junie完成的不好结果需要多次迭代.反而比claude code贵不少.
  
  对于需要设计的任务.我倾向于手写骨架,然后让agent按照骨架完成.(这点待测试)
